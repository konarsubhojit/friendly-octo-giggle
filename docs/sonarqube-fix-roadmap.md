S1082).

| Rule    | Fix                                                                                                         |
| ------- | ----------------------------------------------------------------------------------------------------------- |

## PR-3 — Security Hotspot audit

Two ReDoS hotspots will be reviewed and, per the agreed disposition, marked
**SAFE** with an explanatory comment.

| Hotspot key            | File                                                              | Pattern   |
| ---------------------- | ----------------------------------------------------------------- | --------- |
| `AZ5lNj5qsEsy9yoWZESg` | `src/app/[locale]/(public)/auth/forgot-password/page.tsx:22`      | email RFC |
| `AZ5zooeL4zARl5cMfQwJ` | `src/features/product/components/AssistantMarkdown.tsx:88`        | markdown  |

### Audit checklist (per hotspot)

1. Identify the regex and confirm whether catastrophic backtracking is possible
   on attacker-controlled input.
2. Confirm the regex is bounded by a small max input length or runs against
   trusted/short data.
3. Document findings in the hotspot review comment.
4. If safe, transition the status via the SonarQube MCP tool
   `change_security_hotspot_status` with `status=REVIEWED`,
   `resolution=SAFE`.
5. If not safe, rewrite the regex (anchor, atomic group, or replace with a
   parser) and update tests.

No code change is expected unless an audit reveals a real risk.
